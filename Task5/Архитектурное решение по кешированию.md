# Архитектурное решение по кешированию

## Контекст и цель

MES чувствует себя плохо на чтении dashboard: операторы жалуются, что первая страница долго загружается, хотя уже добавлены фильтры по статусам и пагинация. Операторам важно видеть самые новые заказы, потому что от этого зависит, кто первым возьмет заказ в работу.

Цель решения - ускорить чтение списка заказов в MES dashboard и снизить нагрузку на MES DB, не ломая свежесть статусов и не кешируя данные, которые должны меняться строго консистентно.

## Мотивация

Кеширование нужно не для ускорения всего MES, а для конкретного узкого места: повторного чтения списков заказов по статусам на первой странице dashboard. Эти запросы часто повторяются многими операторами, а результат меняется только при создании заказа или смене статуса.

Кеш должен решить следующие проблемы:

- снизить p95 latency загрузки MES dashboard;
- уменьшить количество одинаковых чтений из MES DB;
- сохранить сортировку newest first для операторов;
- ограничить последствия пропущенной инвалидации коротким TTL;
- дать возможность позже добавить Refresh-Ahead для самых горячих статусов.

В кеширование включаем:

| Элемент | Решение | Причина |
|---|---|---|
| Список заказов по статусу и странице | кешировать | самый частый и дорогой read-path dashboard |
| Первая страница свежих заказов по статусам | кешировать в первую очередь | оператору важны самые новые заказы |
| Детальная карточка заказа | можно кешировать позже с коротким TTL | ниже частота чтения и выше риск устаревших данных |
| Расчет стоимости по 3D-модели | не кешировать как готовый результат без строгого ключа версии файла | расчет зависит от содержимого модели и доменной логики MES |
| Запись изменения статуса | не кешировать | статус заказа должен сначала надежно попасть в MES DB |
| Персональные данные клиента | не кешировать в read model dashboard | лишний риск утечки и сложность очистки |

## Предлагаемое решение

Внедрить серверное кеширование в MES API с Redis. Клиентское кеширование не подходит как основное решение: у разных операторов общий dashboard, данные должны инвалидироваться централизованно при изменении статуса, а браузерный cache не снимет нагрузку с MES API и MES DB для всех пользователей.

Основной паттерн - Cache-Aside для чтения dashboard. MES API сначала проверяет Redis; при cache miss читает MES DB, кладет результат в Redis с коротким TTL и возвращает оператору. Для самой горячей первой страницы можно добавить Refresh-Ahead как оптимизацию второго этапа.

### Ключи и данные

Основной ключ:

```text
orders:status:{status}:page:{page}:sort:newest:v1
```

Примеры:

```text
orders:status:MANUFACTURING_APPROVED:page:1:sort:newest:v1
orders:status:MANUFACTURING_STARTED:page:1:sort:newest:v1
```

Значение в Redis - read model для dashboard: `order_id`, `status`, `created_at`, `updated_at`, `priority`, `operator_id_hash` если заказ уже взят в работу, без PII и без тяжелых вложенных объектов.

TTL:

- 30 секунд для страниц со свежими заказами;
- 60 секунд для менее популярных страниц, если метрики покажут стабильную пользу;
- без бесконечного хранения, потому что пропущенная инвалидация не должна держать устаревший список долго.

### Как работает чтение

1. Оператор открывает MES dashboard со статусом, например `MANUFACTURING_APPROVED`.
2. MES Frontend вызывает `GET /orders?status=MANUFACTURING_APPROVED&page=1`.
3. MES API ищет ключ `orders:status:MANUFACTURING_APPROVED:page:1:sort:newest:v1` в Redis.
4. При cache hit MES API возвращает список из Redis.
5. При cache miss MES API читает MES DB с сортировкой `ORDER BY created_at DESC`, кладет read model в Redis через `SETEX` и возвращает список.

### Как работает запись статуса

1. Оператор берет заказ в работу.
2. MES Frontend вызывает `PATCH /orders/{id}/status`.
3. MES API сначала записывает новый статус в MES DB.
4. После успешной записи MES API удаляет кеш ключей старого и нового статуса для первых страниц.
5. MES API публикует `OrderStatusChanged` в RabbitMQ.
6. Следующее чтение dashboard получит свежий список из DB и заново заполнит Redis.

Если Redis недоступен, MES API должен деградировать в обычное чтение из MES DB: ошибка кеша логируется как WARN, но dashboard продолжает работать. Запись статуса не должна зависеть от Redis.

### Почему Cache-Aside

| Паттерн | Решение |
|---|---|
| Cache-Aside | Подходит: простой контроль в MES API, легко деградировать при недоступном Redis, не требует переписывать весь write-path |
| Write-Through | Хуже: запись статуса должна синхронно обновлять DB и cache, что повышает latency и усложняет rollback |
| Write-Behind | Не подходит: статус заказа нельзя писать в БД отложенно, это риск потери бизнес-события |
| Refresh-Ahead | Подходит только как дополнение для горячих страниц, но не заменяет инвалидацию при смене статуса |
| Client-side cache | Не подходит как основное решение: не решает нагрузку на MES API/MES DB и сложнее централизованно инвалидируется |

## Инвалидация кеша

Выбранная стратегия - комбинированная: программная инвалидация по ключам плюс короткий TTL. Это дает баланс свежести и надежности: штатно данные обновляются сразу после изменения статуса, а TTL ограничивает последствия, если удаление ключа не произошло.

| Стратегия | Подходит | Ограничения |
|---|---|---|
| TTL only | простая защита от бесконечно устаревших данных | оператор может до 30-60 секунд видеть устаревший список после смены статуса |
| Programmatic DEL by key/status | быстро убирает устаревшие read models после записи статуса | нужно аккуратно знать ключи старого и нового статуса |
| Event-based invalidation | полезна, если кеши появятся в нескольких сервисах | асинхронность: событие может задержаться или не обработаться |
| Manual flush | годится только для аварийной поддержки | не подходит как штатная стратегия |

Практическая схема:

- при переходе `old_status -> new_status` удалить ключи первых страниц для `old_status` и `new_status`;
- для горячих статусов хранить небольшой registry ключей, чтобы не использовать дорогой wildcard scan на каждый update;
- после публикации `OrderStatusChanged` дополнительные consumers могут инвалидировать свои read models;
- TTL 30 секунд остается safety net для пропущенной инвалидации;
- при изменении формата read model повысить версию ключа, например `v1 -> v2`.

## Сравнение вариантов

| Критерий | Cache-Aside + Redis + TTL + DEL by status | Refresh-Ahead only | Client-side cache |
|---|---|---|---|
| Свежесть данных | хорошая за счет удаления ключей при записи и короткого TTL | хуже: фоновые jobs могут обновить данные не сразу | зависит от браузера и сложна для централизованной инвалидации |
| Нагрузка на MES DB | сильно ниже на повторных чтениях dashboard | ниже для горячих ключей, но miss-path все равно нужен | почти не снижает нагрузку при разных операторах |
| Сложность внедрения | средняя, изменения в MES API и Redis | выше из-за scheduler/job и критериев популярности ключей | низкая на frontend, но слабый эффект |
| Риск для записи статуса | низкий: DB остается source of truth | низкий, если не вмешиваться в write-path | низкий |
| Поведение при отказе кеша | fallback на MES DB | fallback на MES DB, но jobs будут падать | браузерный cache может отдавать устаревшее |
| Поддержка newest first | хорошая, ключ строится по status/page/sort | хорошая только для заранее выбранных горячих страниц | хуже для общей картины между операторами |

Заключение: основной вариант - Cache-Aside в MES API с Redis, коротким TTL и программной инвалидацией по изменению статуса. Refresh-Ahead стоит добавить позже только для первой страницы самых популярных статусов, если метрики покажут высокий cache miss rate на горячих ключах. Client-side cache можно использовать только как мелкую UX-оптимизацию, но не как архитектурное решение проблемы MES dashboard.

## Метрики внедрения

После запуска кеширования нужно контролировать:

- `mes_dashboard_orders_latency_p95`;
- `redis_cache_hit_ratio` по status/page;
- `mes_db_read_qps` для dashboard queries;
- количество stale reads, найденных через жалобы или проверки;
- ошибки Redis и fallback reads в MES DB;
- время обработки `PATCH /orders/{id}/status`.
