# Планирование: анализ, идентификация проблем и поиск решений

## Контекст

Компания «Александрит» масштабирует B2C и B2B-продажи ювелирных изделий на заказ. Производство готово к двукратному росту, но IT-ландшафт не успевает: заказы задерживаются, теряются между системами, API-партнеры ежедневно жалуются, а операторы MES медленно получают список новых заказов.

## Проблемные места

| Проблема | Где проявляется | Последствие |
| --- | --- | --- |
| Нет сквозной видимости заказа | Shop, CRM, MES, RabbitMQ | Команда не понимает, где заказ завис или потерялся |
| RabbitMQ не контролируется как критический бизнес-поток | CRM API, MES API, очереди | Растут DLQ/in-flight сообщения, но бизнес узнает об этом из жалоб |
| MES API совмещает расчет стоимости, список заказов и работу операторов | MES API, MES db | Медленные расчеты и тяжелые чтения влияют на операторский интерфейс |
| Один инстанс приложения и БД на систему | Shop, CRM, MES, базы данных | Нет запаса отказоустойчивости и масштабирования |
| MES dashboard не оптимизирован под последние заказы | MES, MES API, MES db | Операторы не видят свежие заказы вовремя |
| B2B API открыт без управляемой защиты от нагрузки | API user, MES API | Партнерская нагрузка вытесняет внутренние сценарии |
| Релизы и E2E проверки зависят от ручных шагов | CI/CD, QA | Исправления задерживаются, high/highest баги блокируют прод |
| Логи не дают быстрого разбора инцидентов | Все backend-сервисы | Поддержка восстанавливает картину со слов клиента |

## Инициативы

| Приоритет | Инициатива | Что меняем | Эффект |
| --- | --- | --- | --- |
| 1 | Наблюдаемость order lifecycle | Метрики, логи, трейсы, correlation/order id | Быстро видно, где заказ застрял |
| 2 | Контроль RabbitMQ и партнерского API | DLQ/in-flight alerts, rate limits, backpressure | Снижение потерь и перегрузки |
| 3 | Оптимизация MES dashboard | Read model, кеш, индексы, сортировка по свежести | Операторы быстрее берут новые заказы |
| 4 | Горизонтальное масштабирование API | Несколько инстансов и readiness/liveness | Устойчивость к росту нагрузки |
| 5 | Разделение тяжелых расчетов и пользовательских чтений | Очереди, worker pool, circuit breaker | Расчет цены не блокирует dashboard |
| 6 | Улучшение CI/CD и автотестов | Автоматизация smoke/E2E, quality gates | Меньше задержек релиза |
| 7 | Политики хранения и доступа к observability data | RBAC, masking, retention | Безопасная эксплуатация |

## Целевая архитектура через полгода

Через полгода у «Александрита» должна быть наблюдаемая и управляемая архитектура: все переходы заказа публикуют метрики, структурные логи и трейсы; RabbitMQ контролируется как бизнес-критичная интеграция; MES dashboard читает оптимизированную read model с серверным кешем; партнерский API защищен лимитами, backpressure и circuit breaker; ключевые API запущены минимум в двух инстансах; релизы проходят через автоматизированные проверки.

## Три инициативы на ближайшие полгода

1. Сквозная наблюдаемость order lifecycle.
2. Контроль RabbitMQ и партнерского API.
3. Оптимизация MES dashboard через read model и кеширование.

Эти три пункта выбираются первыми, потому что они напрямую адресуют жалобы клиентов и операторов: показывают, где теряются заказы, защищают систему от всплесков B2B-нагрузки и ускоряют выдачу новых заказов операторам.
